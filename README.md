# 시스템 관제 자동화 스크립트 개발

## 📝 프로젝트 소개
본 프로젝트는 안정적인 서버 운영을 위한 리눅스(Ubuntu) 환경 구축 및 관제 자동화 미션입니다. 
다중 사용자 환경에서의 권한 관리, 네트워크 방화벽 설정, 그리고 시스템 리소스와 애플리케이션의 상태를 모니터링하는 쉘 스크립트(`monitor.sh`) 개발을 통해 인프라 엔지니어링 역량을 확보하는 것을 목표로 합니다.

---

## 실습 환경 구성 및 선정 배경

본 미션은 **MacOS**에서 진행되었으며, 기존의 컨테이너(Docker)나 전통적 가상머신(VM) 대신 **OrbStack (Linux Machines)**을 채택했습니다.

### OrbStack 선정 이유

**1. 컨테이너(Docker) 배제: 과제 본연의 학습 목표 달성**
<br>
<br>
컨테이너는 '1 컨테이너 = 1 프로세스' 구동(단일 책임)을 목적으로 설계된 격리 환경입니다.
<br>
도커는 PID 1(1번 프로세스)를 감시합니다. 만약 PID 2, PID 3 등이 에러가 나더라도 PID 1이 정상적으로 동작한다면 아무런 조치를 취하지 않습니다. 반대로 PID 2, PID 3 등이 정상적으로 동작하더라도 PID 1이 에러가 나면 PID 2, PID 3을 포함한 컨테이너 자체를 종료합니다.
- 리눅스 OS의 경우: systemd라는 특별한 시스템 프로그램이 PID 1이 됩니다. systemd는 시스템 내의 여러 수많은 프로세스들을 감시하고 관리하는 역할을 맡습니다.
반면 컨테이너에서는 실행할 애플리케이션 그 자체가 PID 1이기에 별도의 관리자가 존재하지 않습니다. supervisord라는 가짜 감독관 프로그램을 PID 1로 두는 방법도 있긴 하나 도커의 철학을 거스르는 방식이라 현재는 거의 사용하지 않습니다.
<br>
- Fat Container: 컨테이너를 하나 크게 만들어서 내부에 supervisord 같은 관리자 프로그램을 두고 여러 개의 프로세스를 구동하는 방식으로 도커 초기에 많이 사용됐습니다.
하지만 내부의 핵심 앱이 에러가 발생하고 supervisord가 이를 고치려 작업하고 있을 때, 도커 엔진은 PID 1인 supervisord가 정상적으로 동작하기 때문에 앱으로 계속 사용자 트래픽을 보냅니다. 이로 인해 치명적인 장애가 발생할 수 있습니다.
<br>
- Docker Compose: 여러 개의 컨테이너를 하나의 통합된 서비스로 묶어서 관리할 수 있게 해주는 도구입니다. 컴포즈로 여러 컨테이너를 띄우면 서로 통신할 수 있는 네트워크가 자동으로 구축됩니다.
또한 각 앱들 자체가 PID 1로 동작하기 때문에 에러가 발생하면 해당 컨테이너가 종료됩니다. 도커 엔진은 이를 감지해 재시작 정책에 따라 복구를 수행하고, 외부 로드밸런서를 통해 고장난 컨테이너로 향하는 사용자 트래픽을 차단합니다.
<br>
<br>


이번 미션에서 서비스 데몬, 방화벽, 스케줄러 등 각각의 컨테이너를 따로 만들어서 도커 컴포즈로 묶어도 될 수 있으나, OS 구조의 한계와 클라우드 네이티브 아키텍처의 발전으로 이 방식은 비효율적이게 됩니다.
<br>
<br>
도커는 여러 개의 컨테이너가 있더라도 호스트 컴퓨터의 커널을 같이 공유합니다. 만약 UFW(방화벽) 컨테이너가 방화벽 규칙을 세우러면 호스트의 네트워크 커널(iptables)을 직접 조작해야 합니다.
<br>
이를 허용하려면 해당 컨테이너에 시스템 최고 권한(--privileged, NET_ADMIN)을 줘야 하는데, 이는 격리된 하나의 방에 건물 전체의 열쇠를 주는 것과 같아 큰 보안 취약점이 됩니다. 마찬가지로 서비스 데몬, 스케줄러 등도 강력한 권한이 필요합니다.
<br>
<br>
현대 클라우드 네이티브 환경에서는 인프라 작업을 컨테이너 내부가 아닌 외부(클라우드 제공자 레벨)로 분리하는 방식을 채택했습니다. 
<br>
악성 트래픽을 외부에서 미리 차단하는 것을 통해 순수한 트래픽만 받아 처리할 수 있고, 외부 스케줄링 서비스를 통해 컨테이너 내부에서 항상 cron을 켜두어 메모리를 낭비하는 것을 방지합니다.
<br>
컨테이너는 오직 비즈니스 로직만 실행하는 가볍고 순수한 상태로 유지하고, 네트워크 보안이나 작업 스케줄링 같은 무겁고 위험한 통제 작업은 거대하고 안전한 외부 클라우드 인프라가 책임지도록 맡기는 것입니다.



<br>
<br>

**2. 전통적 VM 대비 효율성**

MacOS에서 전통적 VM을 구동할 경우 리소스(CPU, RAM) 점유율이 높고, 초기 우분투 ISO 설치 및 부팅에 많은 시간이 소요됩니다. 
<br>
<br>
OrbStack은 MacOS에 최적화된 가상 환경 도구입니다. 빠르게 부팅되는 가벼움을 가지면서도, 내부적으로는 `systemd`, `ufw`, `cron`이 완벽히 동작하는 완전한 우분투 OS를 제공합니다. 시스템 제약 없이 인프라 설정 본연의 작업에만 집중할 수 있어 미션에 적합하다 생각했습니다.
